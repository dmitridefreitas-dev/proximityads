P0 — CORRECTNESS BUGS (fix before any of these are recorded/shipped)
================================================================

[P0-1] landscape.html references a timeline key that doesn't exist
  File: two/reel2hook1/landscape.html : ~336
  Code: if(t>=T.vidCheers.s && t<T.freeze.s){...}
  Problem: the landscape T timeline has no `freeze` key (it has vidCheers/textHow/
    rewind/...). Reading T.freeze.s throws a TypeError inside the video-control effect
    on every frame from t=0, so twoguystalking.mov never gets its play/pause logic and
    the whole landscape cut is effectively broken.
  Fix: use the same key as the portrait file — replace `T.freeze.s` with `T.vidCheers.e`.

[P0-2] "three" click() SFX is mis-wired — sub-thump is silent, noise double-routed
  File: three/index.html : ~66-84 (SfxEngine.click)
  Problem: the third layer builds o2 → g3 but g3 is never connected to the graph, so
    the sub-thump is inaudible; meanwhile `g2.connect(c.destination)` runs a second
    time (the line meant to route g3). Also click() routes straight to c.destination,
    bypassing gains.master, so it ignores the master 0.5 trim and is noticeably louder
    than every other cue.
  Fix: connect g3 to the bus (g3.connect(this.gains.master)); delete the duplicate
    g2 connect; route all three layers through this.gains.master, not c.destination.

[P0-3] "one" CTA plays a click + confirm sound with nothing to click
  File: one/index.html : SceneCTA (~384-426) and SFX (~479-480)
  Problem: SFX fires ctaClick @ cta+3.5 and ctaConfirm @ cta+3.8, but SceneCTA renders
    no button-press animation tied to those times (clickTime is computed but the button
    "Browse listings or get matched" never animates). Result: a phantom "tap" sound over
    a static frame.
  Fix: either (a) add the press/scale animation to the CTA button at clickTime (mirror
    the press pattern used in three's "Browse Listings"), or (b) drop the ctaClick/
    ctaConfirm cues. Recommend (a) — a self-clicking CTA reads as a demo of the product.

[P0-4] .mov video assets are a playback risk
  Files: two/reel2hook1/ uses sitting.mov + twoguystalking.mov
  Problem: QuickTime .mov support in browsers is codec-dependent (Chrome often refuses
    H.264-in-MOV, Firefox won't). If it fails the catch path silently leaves a black/
    frozen frame — the whole hook is the video, so the reel opens on black.
  Fix: transcode both to .mp4 (H.264 + AAC) like hook2 already uses; update the <video>
    src. Keep a poster frame so a load failure still shows an image, not black.

[P2-2] In-browser Babel + wall-clock timing can't be reliably exported to MP4
  All reeledits files transform JSX at runtime via @babel/standalone and advance time
  with performance.now(). On any frame drop the animation stutters AND the SFX (scheduled
  on the audio clock) desyncs from the visuals. Screen-recording these gives inconsistent
  takes.
  Options, in order of effort:
    - Short term: add a fixed-timestep option (advance t by 1/30 per rAF) for recording so
      takes are deterministic; record on a fast machine.
    - Right answer: port the finished reels into the existing Remotion project
      (reel5/proximity-reels). Remotion already renders deterministic 1080x1920 MP4s with
      frame-locked Audio — it is the correct delivery path. The reeledits become the
      "design in the browser" step, Remotion the "render" step.

  Files: two/reel2hook3/screenshots/ss1-10.jpg (real Reddit/StudLife pages, see capture.py)
  Problem: shipping real usernames, post text, and other sites' chrome in a paid ad can
    raise privacy/ToS/defamation concerns, and the look is "scraped," not designed.
  Fix: replace with stylized look-alike screenshots (same vibe, fabricated handles/text),

================================================================
P2-6 — DARK MODE VARIANT FOR EVERY REEL
================================================================
Goal: ship a light AND a dark cut of each reeledit (one, hook1, hook2, hook3, three)
so the set can match the destination's look (dark feeds, story backgrounds, A/B tests).
Today every file hard-codes a light palette: offWhite #FAFAFA bg, #0A0A0A text, #FFFFFF
cards. Red (#E23A2F) is the brand constant and stays identical in both modes.

There is already precedent to copy, not invent:
  - reel5/proximity-reels/src/brand.ts has getTheme(isDark) returning a full light/dark
    token set (bg, text, sub, card, cardBorder, fieldBg, reviewBg, glow). Mirror those
    tokens.
  - reel1/reel2 originals are already dark — reuse their bg (#0B0B0C) / paper text feel.

[P2-6a] Tokenize color first (do this WITH P2-1, not after)
  Problem: each file uses a flat `C` object with literal hex (C.offWhite, C.dark, C.white).
    Theming by hand means touching dozens of style props per file x5 files x2 modes.
  Fix: replace the flat C with a theme factory, e.g.:
    const BRAND = { red:'#E23A2F', green:'#22C55E', gray:'#555555' };  // mode-invariant
    const THEME = {
      light: { bg:'#FAFAFA', surface:'#FFFFFF', text:'#0A0A0A', sub:'#555555',
               border:'rgba(0,0,0,.08)', redTint:'#FBE6E4', shadow:'rgba(0,0,0,.1)' },
      dark:  { bg:'#0B0B0C', surface:'#161618', text:'#F5F3EF', sub:'#9A9A9A',
               border:'rgba(255,255,255,.10)', redTint:'rgba(226,58,47,.16)',
               shadow:'rgba(0,0,0,.5)' },
    };
    Read mode from a URL param so one file serves both cuts, no copy:
      const mode = new URLSearchParams(location.search).get('mode')==='dark'?'dark':'light';
      const C = { ...BRAND, ...THEME[mode] };
    Then index.html?mode=dark renders the dark cut; default stays light. This also keeps
    a single source of truth — critical given the reel2 body is shared (see P2-1).

[P2-6b] Token-swap checklist (applies to one, two-body, hook1, hook3, three)
  - background offWhite        -> C.bg
  - text dark                  -> C.text ; secondary gray -> C.sub
  - card/bubble white          -> C.surface
  - hairline borders rgba(0,0,0,..) -> C.border
  - redTint fills (#FBE6E4)    -> C.redTint (dark: low-alpha red)
  - drop shadows               -> deepen in dark (C.shadow)
  - splash + replay screens     -> already dark (#0A0A0B); only flip if a LIGHT splash is
                                   wanted for the light cut (optional).
  - Chrome logo wordmark color -> C.text (currently hard-set to C.dark)
  - Photos, listing images, the red logo, green accents -> UNCHANGED.

[P2-6c] Per-reel special cases (the parts a blind token swap gets wrong)
  one (5 Tabs): the 5 "tab" rows use gray favicon chips (#444-#888) on white cards — in
    dark mode bump chip luminance or add a 1px light ring so they don't vanish into the
    dark card. Review images and their red/green quote callouts are fine as-is.
  hook1 (two-guys + rewind): video + VHS scenes are already dark-neutral; only the body
    needs tokens. The grey-hold after the rewind reads in both modes.
  hook2 (glass shatter): the shatter plays on a WHITE frame and "is broken." text sits on
    it. In dark mode flip that frame to near-black (#0B0B0C) and confirm glass.mp4 reads
    on black (a shatter shot on white may look wrong inverted — may need a dark-bg encode
    of the clip, or keep the shatter frame white in BOTH modes as an intentional flash).
    Decide per art direction; do not auto-invert the video.
  hook3 (screenshot pile-on): the pile bg (offWhite) -> C.bg; the polaroid card stock
    stays light/white (real screenshots are light) so they pop against a dark pile — good.
    The "Meet Proximity." beat already cuts to pure black, so it needs no change.
  three: browser-chaos window is a skeuomorphic light Chrome window — keep the browser
    LIGHT in both modes (a dark browser would muddy the "tab chaos" read); only theme the
    Proximity intro, the Five-Nine listing card, and the CTA. The brookings hero is a
    photo, mode-invariant.

[P2-6d] Build/deliver both cuts
  - With the ?mode= approach, no file duplication: record each reel twice
    (index.html and index.html?mode=dark).
  - Update reeledits/../index.html launcher (root) to list a Light/Dark toggle or two
    links per reel.
  - If/when ported to Remotion (P2-2), this maps 1:1 onto reel5's existing dark/light
    Composition pair — getTheme(isDark) is the same idea, so do the tokenization now and
    the Remotion port inherits it for free.

Effort: ~1 file of token plumbing per reel (or once, in the shared body from P2-1), plus
the 5 special-case decisions in P2-6c. Doing P2-1 (de-dup) and P2-6a (tokenize) together
is the efficient path — both are "stop hard-coding, centralize" refactors.
